Avoid Fragmenting Our Efforts

More generally, stop clients from dictating unscalable solutions because they have no experience of scale whereas we do. Scale always magnifies the slightest inconsistencies and errors which are invisible at the small scale normally encountered in a customer's daily life. These invisible problems become highly visible and thus problematic to us when we are working on lots of their files at once.

Conversely, it is much more profitable to work on big projects than small ones!

Dividing one big potentially profitable project into 4 small projects potentially produces 4 unprofitable projects because of all the task switching with other projects going on simultaneously.

The ideal conversion

(1) We clean up the customers data

(2) We do a proof of concept for the customer using 1 or 2 representative documents from each author (and their style guide).

(3) While we do the PoC the customer practices taking snapshots of all their content. When the PoC is done the customer sends us a snapshot of ALL their content.

(4) We enhance the conversion program and show the customer the proposed results applied to their test snapshot.

(5) When the customer reasonably happy with the results they take a new production snapshot - preferably on Friday and send it to us.

(7) We run the conversion program and send the results back within a few hours.

(8) The customer checks the results and if good accepts them into their production system and starts using them on Monday.

(9) If the customer discovers more changes they would like to make they should: (a) Send us a new snap shot and a description of the changes to be made (b) We write/test/run the code to perform the requested change (c) If the customer is using a Git based system we send them the changed files as new branches and the authors merge the branches as they see fit. (d) If the customer is not using a Git based system we send them differential patches to apply as they see fit. This is identical to the process that Git uses when merging two branches so there is nothing novel here. (e) Repeat as necessary until done.

Ideally we would bill after each step and not advance to the next step until we are confident with the customers willingness to pay for the work completed so far.

This approach allows us to get the maximum benefit from the conversion effort in the shortest amount of time. Alternative solutions divide the work up too much and spend too much time redoing the conversion to accommodate trailing minor changes much better done as described in (8).

A major benefit of organizing work in this manner is that the hard part (the conversion) can be done in one sprint without swapping to other projects because all the data is available.

The subsequent easier incremental work can be done in short self contained sprints gain without swapping.
